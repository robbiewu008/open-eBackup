Step 5: Creating a Backup SLA

The system provides three preset Service Level Agreements (SLAs): Gold, Silver, and Bronze. If the preset SLAs meet your backup requirements, skip this section. You can also customize SLAs based on your requirements to protect resources.

Procedure

  1. Choose Protection > Protection Policies > SLAs.
  2. Click Create.
  3. Enter an SLA name.
  4. Select an application type for the SLA to be created.

    1. Click Applications and select the application for the SLA to be created.
      • General SLA

        A general SLA can be applied to resources except filesets, SQL Server, PostgreSQL, Kingbase, Redis, ClickHouse, MongoDB, NAS shares, NAS file systems, Huawei Cloud Stack GaussDB, TDSQL, Kubernetes CSI, TiDB, volumes, common shares, Active Directory, GaussDB T single-node systems, object storage, GaussDB (1.6.0 and later versions), and SAP HANA (as an application) (1.6.0 and later versions).

      • Specific to application

        Select an application type for the SLA to be created. This section uses Specific to application as an example.

    2. Click OK.

  5. Configure a backup policy.

    1. Click the Backup Policy icon.
    2. Set basic parameters for the backup policy.

      Table 1 describes related parameters.

      Set the backup interval, copy retention period, and backup time window based on service requirements. The recommended settings are as follows:
      • The backup interval must be longer than the backup duration.
      • The retention period of a full backup copy must be longer than that of an incremental backup (or forever incremental (synthetic full) backup) copy.
      • The retention period must be longer than the backup interval.
      • At least one backup policy must be created. To add another backup policy, click on the right of the page.
      • A maximum of four backup policies can be added for each backup type.
      Table 1 Basic parameters of a backup policy

      Parameter

      Description

      Full backup

      Name

      Name of a backup policy.

      -
      Configure the backup interval, copy retention period, and backup time window for full backup.
      • Configure the backup interval and copy retention period for full backup.
        • By Year

          Configure the job to be executed once every year on xx (month) xx (day). Configure copies to be retained for xx days, weeks, months, years, or permanently.

          If the date does not exist in the year, no copy is generated.

        • By Month

          Configure the job to be executed on xx (day) (multiple days can be selected) or the last day of each month. Configure copies to be retained for xx days, weeks, months, years, or permanently.

          If the backup job is set to be performed on the xxth day of each month, no copy is generated when the date does not exist in the current month.

        • By Week

          Configure the job to be executed every Monday, Tuesday, Wednesday, Thursday, Friday, Saturday, or Sunday (you can select multiple options). Configure copies to be retained for xx days, weeks, months, years, or permanently.

        • By Day

          Configure the job to be executed every xx days starting from xx (year) xx (month) xx (day). Configure copies to be retained for xx days, weeks, months, years, or permanently.

        • By Hour

          Configure the job to be executed every xx hours starting from xx (year) xx (month) xx (day). Configure copies to be retained for xx days, weeks, months, years, or permanently.

      • Set the time period for performing full backup. The full backup job will not be scheduled beyond the time range.
        NOTE:
        • If the end time is earlier than or the same as the start time, the end time is actually the end time of the next day.
        • If the backup job is not completed within the specified time window, the system does not stop the backup job, but reports an event.
        • Once the retention period expires, the system automatically deletes the expired copies.

      Incremental backup

      Name

      Name of a backup policy.

      NOTE:

      Incremental protection policies can be created only for database instances.

      -
      Configure the backup interval, copy retention period, and backup time window for incremental backup. Incremental backup and differential backup cannot coexist.
      • Configure the backup interval and copy retention period for incremental backup.
        • By Year

          Configure the job to be executed once every year on xx (month) xx (day). Configure copies to be retained for xx days, weeks, months, years, or permanently.

          If the date does not exist in the year, no copy is generated.

        • By Month

          Configure the job to be executed on xx (day) (multiple days can be selected) or the last day of each month. Configure copies to be retained for xx days, weeks, months, years, or permanently.

          If the backup job is set to be performed on the xxth day of each month, no copy is generated when the date does not exist in the current month.

        • By Week

          Configure the job to be executed every Monday, Tuesday, Wednesday, Thursday, Friday, Saturday, or Sunday (you can select multiple options). Configure copies to be retained for xx days, weeks, months, years, or permanently.

        • By Day

          Configure the job to be executed every xx days starting from xx (year) xx (month) xx (day). Configure copies to be retained for xx days, weeks, months, years, or permanently.

        • By Hour

          Configure the job to be executed every xx hours starting from xx (year) xx (month) xx (day). Configure copies to be retained for xx days, weeks, months, years, or permanently.

        • By Minute

          Configure the job to be executed every xx minutes starting from xx (year) xx (month) xx (day). Configure copies to be retained for xx days, weeks, months, years, or permanently.

      • Set the time period for performing incremental backup. The incremental backup job will not be scheduled beyond the time range.
        NOTE:
        • If the end time is earlier than or the same as the start time, the end time is actually the end time of the next day.
        • If the backup job is not completed within the specified time window, the system does not stop the backup job, but reports an event.
        • Once the retention period expires, the system automatically deletes the expired copies.

      Log Backup

      Name

      Set the backup policy name.

      NOTE:

      Only CMDB supports log backup.

      -

      Back up the archive logs of a CMDB database. Configure the initial backup time, backup interval, and copy retention period for log backup.

      • By Day

        Configure the job to be executed every xx days starting from xx (year) xx (month) xx (day). Configure copies to be retained for xx days, weeks, months, years, or permanently.

      • By Hour

        Configure the job to be executed every xx hours starting from xx (year) xx (month) xx (day). Configure copies to be retained for xx days, weeks, months, years, or permanently.

      • By Minute

        Configure the job to be executed every xx minutes starting from xx (year) xx (month) xx (day). Configure copies to be retained for xx days, weeks, months, years, or permanently.

      NOTE:
      • The log backup interval must be less than the data backup interval.
      • After a database is restored, the next log backup will back up all logs generated after the restoration time point.
      • If a resource is associated with an SLA configured with log backup and then associated with another SLA that is not configured with log backup, the archive logs that have been backed up are deleted according to the previously defined archive log retention policy.
      • You can specify a time point only after backing up archive logs.
      • If restoration has been performed, ensure that a new full copy has been generated after the last restoration before the next log backup. You are advised to manually perform a full backup. Otherwise, the log backup will fail.
      • Log backup policies cannot be configured separately.
      • If log copies need to be continuous in time, ensure that archive logs generated between the two log backups are not cleaned.
      • Before backing up logs, ensure that a full backup copy exists.
    3. For 1.6.0 and later versions: To generate a WORM copy, enable WORM and set the WORM validity period. Otherwise, skip this step.
      For WORM Validity Period, you can select Same as the copy retention period or Custom validity period.
      • WORM copies cannot be deleted or their validity periods cannot be shortened.
      • The custom WORM validity period cannot be longer than the copy retention period.
      • A WORM copy cannot be changed to a non-WORM copy within its validity period.
    1. Set advanced parameters for the backup policy.
      Table 2 describes related parameters.
      Table 2 Advanced parameters of a backup policy

      Parameter

      Description
      Specify Target Location
      NOTE:

      This parameter is supported only by the X series backup appliances.
      Specify the target storage location of backup copies.
      • Backup storage unit: corresponds to a backup cluster node. After a backup storage unit is selected, copies are stored to the storage unit. If the capacity of the storage unit is insufficient, the backup job will fail.
      • Backup storage unit group: is a collection of backup storage units. After a backup storage unit group is selected, the system automatically selects a target backup storage unit based on the storage policy of the storage unit group.
      NOTE:
      • For 1.5.0, this parameter is displayed only when you have deployed the cluster HA feature. You can leave the target location unspecified. If you do not specify the target location, the system automatically selects the target storage unit through intelligent balancing.
      • For 1.6.0 and later versions, you must specify the target location.

      Rate Limiting Policies

      Average maximum amount of data that can be transmitted per second in a backup job.

      Choose a rate limiting policy created in Step 3: Creating a Rate Limiting Policy.
      Automatic Indexing
      NOTE:

      This parameter is available only in 1.5.0.
      If the disks to be backed up contain system disks and this function is enabled, you can select one or more files in the copy for restoration and search for files, directories, or links in the copy.
      NOTE:

      Indexes occupy backup storage space. If the number of index files in the system exceeds 1 billion, perform operations by referring to "Failed to Create a Copy Index" in the OceanProtect Appliance 1.5.0-1.6.0 Troubleshooting.

      Concurrent Jobs

      Number of backup jobs that can be executed simultaneously.

      Standby Node Preferred for Backup

      After this function is enabled, the standby node is preferentially used for backup.

      Source Deduplication
      After this function is enabled, the amount of data to be transmitted can be reduced, but more client resources will be occupied.
      NOTE:
      • An agent host can only be linked with one storage unit. To enable source deduplication, select Backup Storage Unit for Specify Target Location. Otherwise, a storage unit that is not linked with the agent host may be selected through intelligent balancing. As a result, source deduplication fails to be enabled. If different protected objects are associated with different SLAs, where different backup storage units are specified, select different agent hosts for the protected objects. Otherwise, source deduplication of some backup jobs will fail to be enabled.
      • If an agent host has been linked with a storage unit but is not performing a backup job with source deduplication, you can run the dataturbo delete storage_object storage_name=? command on the agent host to delete the source deduplication link with the storage unit. The value of storage_name in the command can be queried by running the dataturbo show storage_object command. This way, when the agent host performs a backup job with source deduplication, the link between the agent host and a new storage unit can be established.
      • Check whether the host OS supports source deduplication by referring to OceanProtect Compatibility Query. If the host OS does not support source deduplication, source deduplication does not take effect after being configured.
      • If a built-in agent (that is, ProtectAgent is co-deployed on the OceanProtect) is used for backup, source deduplication does not take effect after being configured.
      Job Timeout Alarm
      NOTE:

      This parameter is available only in 1.6.0 and later versions.

      After this function is enabled, an alarm is sent if the job execution time exceeds the time window. The alarm needs to be manually cleared.

      Job Failure Alarm

      After this function is enabled, an alarm is automatically sent when a job fails. The alarm is automatically cleared when the next job is successful.

      Automatic Retry

      After this function is enabled, the system automatically retries a backup job if it fails.

      The number of retries ranges from 1 to 5, and the interval ranges from 1 to 30 minutes. For example, if the number of retries is set to 3 and the interval to 5 minutes, the system makes a total of three retries, with an interval of 5 minutes between each retry.
      NOTE:

      During automatic retry, the system creates a backup job. If the backup job is not within the backup time window, the job is not executed and fails.
    2. Click OK.

  6. Click OK.
