Managing MongoDB Copies

This section describes the operations that you can perform on copy data on the product.

Log in to the management page and choose Explore > Copy Data > Big Data > MongoDB to go to the MongoDB copy page.

  • Click Resources to perform operations by resource. Operations on this page are performed on replication copies of the current resource. Perform the following operations in the cluster where the replication copies reside.

    Table 1 describes the related operations.

    Table 1 Operations related to replication copies

    Operation

    Description

    Navigation Path

    Creating protection for replication copies

    Scenario

    This operation enables you to periodically replicate or archive replication copies.

    Note

    If no replication copy is generated for the selected resource, you cannot associate an SLA with the resource for protection.

    In the row of the target resource, choose More > Protect, and associate a replication SLA or an archive SLA to replicate or archive the replication copy.

    Modifying the SLA associated with replication copies

    Scenario

    This operation enables you to modify the SLA associated with replication copies.

    In the row of the target resource, choose More > Modify Protection to associate a new SLA with the replication copy.

    Removing protection for replication copies

    Scenario

    This operation enables you to remove the protection plan if the replication copies no longer need to be protected. After removal, the system cannot protect these replication copies.

    In the row of the target resource, choose More > Remove Protection.

    Activating protection for replication copies

    Scenario

    This operation enables you to activate the protection plan for replication copies. After the protection plan is activated, the system protects replication copies based on the protection plan.

    Choose More > Activate Protection.

    Disabling protection for replication copies

    Scenario

    This operation enables you to disable the protection plan if the replication copies no longer require periodic protection. After the protection plan is disabled, the system does not execute automatic protection jobs of these replication copies.

    Choose More > Disable Protection.

    Performing manual replication for replication copies

    Scenario

    This operation enables you to replicate a replication copy immediately.

    Note

    Before performing this operation, ensure that a replication copy-based SLA has been created for the current resource and the SLA is a replication SLA.

    Choose More > Manually Replicate.
    Modifying the owner
    NOTE:

    Only 1.6.0 and later versions support this function.

    Scenario

    This operation enables you to modify the owner of a resource and all its copies and switch the quota occupied by the copies to a new owner.

    Note
    • Only the system administrator has the permission to modify the owner.
    • The owner can be modified only to a user of the Data Protection Administrator or custom role.
    • After the owner of a resource is modified, all information about the resource will not be displayed under the original owner.

    In the row of the target resource, choose More > Modify Owner.
  • Click Copy to perform operations by copy.

    Table 2 describes the related operations.

    Table 2 Operations related to MongoDB copies

    Operation

    Description

    Navigation Path
    Performing manual archiving
    NOTE:

    Only 1.6.0 and later versions support this function.

    Scenario

    This operation enables you to manually archive copies to the archive storage.

    Note

    Only full backup copies and forever incremental (synthetic full) backup copies, as well as their replication copies and cascaded replication copies can be manually archived.

    In the row of the target copy, choose More > Manual Archive.

    Modifying a copy retention policy

    Scenario

    This operation enables you to modify the retention period of a copy.

    Note

    • The modified copy retention period is calculated once the modification takes effect.
    • When you change the retention period of a full copy, the retention period of the full copy cannot be shorter than that of the incremental copy generated when the incremental backup is performed based on the full copy.

    In the row of the target copy, choose More > Modify Retention Policy.

    You can set the retention period to be forever or to xx days, xx weeks, xx months, or xx years.

    Deleting a copy

    Scenario

    This operation enables you to delete a copy to free up storage space if it is not required or becomes invalid.

    Note

    • If a resource has multiple full copies, the latest full copy cannot be deleted.
    • If a resource has multiple copies and only one full backup copy, the full backup copy can be deleted.
    • If a resource has multiple copies and only one full replication copy, the full replication copy cannot be deleted.
    • If a full copy is deleted, the system automatically deletes the incremental copies that depend on the full copy.
    In the row of the target copy, choose More > Delete. Alternatively, select the copies to be deleted and click Delete above the table.
    NOTE:

    For 1.6.0 and later versions, to forcibly delete the copy data in the database after copy deletion fails, select Forcibly delete the copy data upon deletion failure in the displayed dialog box.

    Executing reverse replication or cascaded replication on replication copies

    Scenario

    This operation enables you to replicate a replication copy to the source end for data restoration or replicate a replication copy to another cluster for geographic redundancy.

    Note

    Before performing reverse replication, delete the backup copies corresponding to the replication copies from the source end. Otherwise, the replication will fail.
    In the row of the target replication copy, choose More > Replicate. Configure copy replication parameters.
    1. Select Replication Target Cluster.
      • If Replication Target Cluster is the source cluster, reverse replication is performed.
      • If Replication Target Cluster is another cluster, cascaded replication is performed.
    2. For 1.6.0 and later versions, select Replication Target User.
    3. For 1.6.0 and later versions, enter the password of the replication target user and click Authenticate.
    4. Set Specify Target Location for replication.
    5. Set the copy retention period.
    6. Enable or disable deduplication and compression.

      You are advised to enable these functions to improve replication efficiency.

    7. Enable or disable the job failure alarm function.

    Note: The configuration for restoring a database is similar to that in Restoring MongoDB and is not described in this section.
