Step 5: Creating a Backup SLA

The system provides three preset SLAs: Gold, Silver, and Bronze. If the preset SLAs meet your backup requirements, skip this section. You can also customize SLAs based on your requirements to protect resources.

Context

For details about preset SLAs and custom SLAs, see About SLA.

Procedure

  1. Choose Protection > Protection Policies > SLAs.
  2. Click Create.
  3. Enter an SLA name.
  4. Select an application type for the SLA to be created.

    1. Click Applications and select the application for the SLA to be created.
      • General SLA

        A general SLA can be applied to resources except filesets, SQL Server, PostgreSQL, Kingbase, Redis, ClickHouse, MongoDB, NAS shares, NAS file systems, Huawei Cloud Stack GaussDB, TDSQL, Kubernetes CSI, TiDB, volumes, common shares, GaussDB T single-node systems, and object storage.

      • Specific to application

        Select an application type for the SLA to be created. This section uses Specific to application as an example.

    2. Click OK.

  5. Configure a backup policy.

    1. Click the Backup Policy icon.
    2. Set basic parameters for the backup policy.

      Table 1 describes the related parameters.

      Set the backup interval, backup retention period, and backup time window based on service requirements. The recommended settings are as follows:
      • The backup interval must be longer than the backup duration.
      • The retention period of a full backup copy must be longer than that of an incremental backup (or forever incremental (synthetic full) backup) copy.
      • The retention period must be longer than the backup interval.
      • At least one backup policy must be created. To add another backup policy, click on the right of the page.
      • A maximum of four backup policies can be added for each backup type.
      Table 1 Basic parameters of a backup policy

      Parameter

      Description

      Full backup

      Name

      Name of a backup policy.

      -
      Configure the execution time, backup interval, copy retention period, and backup time window for the full backup.
      • Configure the backup interval and copy retention period for full backup.
        • By Year

          Configure the job to be executed once every year on xx (month) xx (day). Configure copies to be retained for xx days, weeks, months, years, or permanently.

          If the date does not exist in the year, no copy is generated.

        • By Month

          Configure the job to be executed on xx (day) (multiple days can be selected) or the last day of each month. Configure copies to be retained for xx days, weeks, months, years, or permanently.

          If the backup job is set to be performed on the xxth day of each month, no copy is generated when the date does not exist in the current month.

        • By Week

          Configure the job to be executed every Monday, Tuesday, Wednesday, Thursday, Friday, Saturday, or Sunday (you can select multiple options). Configure copies to be retained for xx days, weeks, months, years, or permanently.

        • By Day

          Configure the job to be executed every xx days starting from xx (year) xx (month) xx (day). Configure copies to be retained for xx days, weeks, months, years, or permanently.

        • By Hour

          Configure the job to be executed every xx hours starting from xx (year) xx (month) xx (day). Configure copies to be retained for xx days, weeks, months, years, or permanently.

      • Set the time period for performing full backup. The full backup job will not be scheduled beyond the time range.
        NOTE:
        • If the end time is earlier than or the same as the start time, the end time is actually the end time of the next day.
        • If the backup job is not completed within the specified time window, the system does not stop the backup job, but reports an event.
        • Once the retention period expires, the system automatically deletes the expired copies.

      Forever Incremental (Synthetic Full) Backup

      Name

      Name of a backup policy.

      -
      Configure the first execution time, backup interval, copy retention period, and backup time window for forever incremental (synthetic full) backup. The initial backup is a full backup by default.
      • Configure the backup frequency and copy retention period for forever incremental (synthetic full) backup.
        • By Year

          Configure the job to be executed once every year on xx (month) xx (day). Set copies to be retained for xx days, weeks, months, years, or permanently.

          If the date does not exist in the year, no copy is generated.

        • By Month

          Configure the job to be executed on xx (day) (multiple days can be selected) or the last day of each month. Set copies to be retained for xx days, weeks, months, years, or permanently.

          If the backup job is set to be executed on the xxth day of each month, no copy is generated when the date does not exist in the current month.

        • By Week

          Configure the job to be executed every Monday, Tuesday, Wednesday, Thursday, Friday, Saturday, or Sunday (you can select multiple options). Set copies to be retained for xx days, weeks, months, years, or permanently.

        • By Day

          Configure the job to be executed every xx days starting from xx (year) xx (month) xx (day). Configure copies to be retained for xx days, weeks, months, years, or permanently.

        • By Hour

          Configure the job to be executed every xx hours starting from xx (year) xx (month) xx (day). Configure copies to be retained for xx days, weeks, months, years, or permanently.

        • By Minute

          Configure the job to be executed every xx minutes starting from xx (year) xx (month) xx (day). Configure copies to be retained for xx days, weeks, months, years, or permanently.

      • Set the allowed time period for executing Forever Incremental (Synthetic Full) Backup. Forever Incremental (Synthetic Full) Backup jobs will not be scheduled at the time beyond the time period.
        NOTE:
        • If the end time is earlier than or the same as the start time, the end time is actually the end time of the next day.
        • If the backup job is not completed within the specified time window, the system does not stop the backup job, but reports an event.
        • Once the retention period expires, the system automatically deletes the expired copies.
    3. Set advanced parameters for the backup policy.
      Table 2 describes the related parameters.
      Table 2 Advanced parameters of a backup policy

      Parameter

      Description

      Specify Target Location
      If you have deployed the cluster HA feature, specify the target storage location for the backup copy. If no target location is specified, the system automatically selects a backup target storage unit through intelligent balancing.
      • Backup storage unit: corresponds to a backup cluster node. After a backup storage unit is selected, copies are stored to the storage unit. If the capacity of the storage unit is insufficient, the backup job will fail.
      • Backup storage unit group: is a collection of backup storage units. After a backup storage unit group is selected, the system automatically selects a target backup storage unit based on the storage policy of the storage unit group.

      Rate Limiting Policies

      Maximum amount of data that can be transmitted by a backup job per second.

      Select a created rate limiting policy.

      Verify Copy

      After the backup is complete, the system will automatically perform a copy verification to verify the integrity of the copy, you can also manually perform copy verification. Backup and restoration performance will be affected.

      If the copy verification fails after the backup is complete, the next backup job is a full backup job.

      NOTE:

      If a backup job has been performed for the resource associated with the SLA for which copy verification is disabled, after the copy verification function is enabled, the initial backup job is converted to a full backup job if it is a non-full backup job. The possible scenarios are detailed as follows:

      • The resource object is associated with an SLA for which copy verification is disabled, and the SLA is modified later with the copy verification function enabled.
      • The resource object is associated with an SLA for which copy verification is disabled and is subsequently associated with other SLAs (the copy verification function has been enabled).

      Source Deduplication
      After this function is enabled, the amount of data to be transmitted can be reduced, but more client resources will be occupied.
      NOTE:
      • An agent host can only be linked with one storage unit. To enable source deduplication, select Backup Storage Unit for Specify Target Location. Otherwise, a storage unit that is not linked with the agent host may be selected through intelligent balancing. As a result, source deduplication fails to be enabled. If different protected objects are associated with different SLAs, where different backup storage units are specified, select different agent hosts for the protected objects. Otherwise, source deduplication of some backup jobs will fail to be enabled.
      • If an agent host has been linked with a storage unit but is not performing a backup job with source deduplication, you can run the dataturbo delete storage_object storage_name=? command on the agent host to delete the source deduplication link with the storage unit. The value of storage_name in the command can be queried by running the dataturbo show storage_object command. This way, when the agent host performs a backup job with source deduplication, the link between the agent host and a new storage unit can be established.
      • Check whether the host OS supports source deduplication by referring to Compatibility Query. If the host OS does not support source deduplication, source deduplication does not take effect.
      • If a built-in agent is used for backup, source deduplication, after being configured, does not take effect.
      Job Timeout Alarm
      NOTE:

      This parameter is available only in 1.6.0 and later versions.

      After this function is enabled, an alarm is sent if the job execution time exceeds the time window. The alarm needs to be manually cleared.

      Job Failure Alarm

      After this function is enabled, an alarm is automatically sent if a job fails and is automatically cleared after successful execution.

      Automatic Retry

      After this function is enabled, the system automatically retries if a backup job fails.

      The number of retries ranges from 1 to 5, and the waiting time ranges from 1 to 30 minutes. For example, if the number of retries is set to 3 and the waiting time to 5 minutes, the system makes a total of three retries and performs each retry every 5 minutes.
      NOTE:

      During automatic retry, the system creates a backup job. The backup job is executed only when it is within the preset time window.
    4. Click OK.

  6. (Optional) Configure an archive policy.

    The built-in storage space of the OceanProtect is limited. You are advised to archive copy data to external storage for long-term retention.

    You can configure an archive policy by clicking Archive Policy or by modifying the SLA.

    For details about how to configure an archive policy, see Archiving.

  7. (Optional) Configure a replication policy.

    The OceanProtect allows you to replicate local data to a remote data center. In case a disaster occurs, copy data in the remote data center can be used to restore production data and services can be taken over, achieving remote data protection.

    You can configure a replication policy by clicking Replication Policy or by modifying the SLA.

    For details about how to configure a replication policy, see Replication.

  8. Click OK.
