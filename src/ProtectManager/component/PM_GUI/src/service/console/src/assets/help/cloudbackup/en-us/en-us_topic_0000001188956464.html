Step 3: Creating a Backup SLA

You need to create a backup SLA to back up file systems.

Precautions

  • Invalid soft link files cannot be backed up.
  • Files of non-common types, such as pipe and socket, cannot be backed up.
  • The backup window of the secondary end of asynchronous replication must be different from the replication window. If the policy is configured in waiting mode, the waiting time after replication is complete must be longer than 3 minutes.
  • File systems whose names start with pvc_ cannot be backed up.
  • Protection cannot be added to the secondary end of a synchronous remote replication pair.
  • File systems in WORM, CLONE, and non-UNIX security modes cannot be backed up.
  • After HyperMetro brain-split occurs, the end with failed arbitration does not support backup and restore.
  • Only the file system of the default tenant System_vStore can be backed up.
  • Only NFS can be used for backup. CIFS cannot be used.

Procedure

  1. Choose Protection > Protection Policies > SLAs.
  2. Click Create.
  3. Enter an SLA name.
  4. Select an application type for the SLA to be created.

    1. Click Applications. In the displayed window, select Local File System.
    2. Click OK.

  5. Click Backup Policy.
  6. Set basic parameters for the backup policy.

    1. Select the backup storage created in section 3.1.1 from the Backup Storage drop-down list on the right.
    2. Set incremental backup parameters. Two backup modes are supported: periodic backup and backup at a specified time.

      Table 1 describes the related parameters.

      Set the backup interval, backup retention period, and backup time window based on service requirements. The recommended settings are as follows:
      • If the file system to be protected belongs to the secondary end of a remote replication pair, ensure that the backup window of the storage system is different from the replication window of the remote replication pair. Otherwise, the backup job may fail.
      • The backup interval must be longer than the backup duration.
      • The execution time of the initial backup should be the same as the start time of the backup time window.
      • The retention period must be longer than the backup interval.
      Table 1 Basic parameters of a backup policy

      Parameter

      Description

      Name

      Set the name of the backup policy.

      -
      Configure the initial execution time, backup interval, copy retention period, and backup time window for incremental backup.
      • Configure the backup frequency and copy retention period for incremental backup.
        • By Year

          Configure the job to be executed once every year on XX (month) XX (day), and copies are retained for XX days, weeks, months, years, or permanently.

          If the date does not exist in the year, no copy is generated.

        • By Month

          Configure the job to be executed on XXth (day) (multiple days can be selected) or the last day of each month, and copies are retained for XX days, weeks, months, years, or permanently.

          If the backup job is set to be performed on the XXth day of each month, no copy is generated when the date does not exist in the current month.

        • By Week

          Set the job to be executed once every Monday, Tuesday, Wednesday, Thursday, Friday, Saturday, or Sunday (you can select multiple options). Copies are retained for XX days, weeks, months, years, or permanently.

        • By day

          Set the backup job to be performed every xx days starting from xx (year) xx (month) xx (day). Copies are retained for xx days, xx weeks, xx months, xx years, or permanently.

        • By Hour

          Set the backup job to be performed every xx hours starting from xx (year) xx (month) xx (day). Copies are retained for xx days, xx weeks, xx months, xx years, or permanently.

        • By Minute

          Set the backup to be performed every xx minutes starting from xx (year) xx (month) xx (day). Copies are retained for xx days, xx weeks, xx months, xx years, or permanently.

      • Set the time period for performing incremental backup. The incremental backup job will not be scheduled beyond the time range.
        NOTE:
        • If the end time is earlier than or the same as the start time, the end time is actually the end time of the next day.
        • If a backup job is not completed within the specified time window, the system does not stop the backup job, but reports an alarm.
        • Once the retention period expires, the system automatically deletes the expired copies.
    3. To add an incremental backup plan, click Add Policy on the right.

      Each backup storage can be associated with only one backup policy.

  7. Set advanced parameters for the backup policy.

    Table 2 describes the related parameters.
    Table 2 Advanced parameters of a backup policy

    Parameter

    Description

    Rate Limiting Policies

    Maximum amount of data that can be transmitted by a backup job per second.

    Select the rate limiting policy created in Step 2: Creating a Rate Limiting Policy.

    Automatic Indexing

    Determines whether to enable automatic indexing. After automatic indexing is enabled, the system automatically creates indexes for files or directories in copies and performs file-level restore and download operations on the files or directories.

    Aggregate

    Determines whether to aggregate multiple small files into a large file to improve the backup speed.

    Network Acceleration

    Determines whether to enable network acceleration. After this function is enabled, Bottleneck Bandwidth and Round-trip propagation time (BBR) network acceleration can be used. You are advised to enable this function when the network latency is greater than or equal to 20 ms.

    NOTE:

    You are advised to set the Network Acceleration parameter to the same value for different buckets of the same OBS. That is, you can enable or disable network acceleration for all the buckets at the same time.

    Automatic Retry

    After this function is enabled, the system automatically retries in case a backup job fails.

    The number of retries ranges from 1 to 5, and the waiting time ranges from 1 to 30 minutes. For example, if the number of retries is set to 3 and the waiting time to 5 minutes, the system makes a total of three retries and performs each retry every 5 minutes.
    NOTE:

    During automatic retry, the system creates a backup job. The backup job is executed only when it is within the preset time window.

    Synthesis Full Cycle

    Specifies the number of backup times before a full backup. For example, if you enable Synthesis Full Cycle and set Backup to 5, the system performs full backup at the sixth backup.

  8. Click OK. The system returns to the page for creating an SLA.
  9. Click OK. The SLA is created.
