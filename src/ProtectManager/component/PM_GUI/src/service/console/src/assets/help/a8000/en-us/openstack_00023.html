Step 12: Performing Backup

Before performing backup, you need to associate the resources to be protected with a specified SLA policy. The system protects the resources based on the SLA policy and periodically performs backup jobs based on the SLA policy. You can perform a backup job immediately through manual backup.

Precautions

  • A cloud server can be backed up only when it is in the running, stopped, or suspended state.
  • A maximum of 24 disks can be mounted to an agent host. Therefore, if 24 disks have been mounted to an agent host during backup or restoration, the system performs the backup job when there are idle slots.

Context

  • During manual backup, parameters (except Automatic Retry) defined in the SLA are applied to the manual backup job.
  • Copies generated by manual backup are retained for the period defined in the SLA.

    If the protection policy selected during manual backup is not defined in the SLA, the generated backup copies will be retained permanently. If full backup is not defined in the SLA but selected during manual backup, the copies generated by the manual full backup are retained permanently.

  • If a replication or archive policy has been defined in the SLA, the system will perform replication or archiving once based on the SLA when you perform manual backup.

Procedure

  1. Choose Protection > Cloud Platforms > OpenStack.
  2. Periodically back up a project.

    1. Click the Project tab and select the project to be protected.
    2. Click Protect.

      The system backs up all cloud servers in the project.

    3. Select a preset or custom SLA.

      You can also click Create to create an SLA.

      If a WORM policy has been configured for the resource to be protected, select an SLA without a WORM policy to create protection to avoid WORM policy conflicts.

    4. Select an agent host.

      If the agent host is not selected, the system performs backup by selecting one or multiple agent hosts from the agent hosts configured during resource registration.

    5. Set the parameters as follows:
      1. Set SLA Overwrite Policy.
        The default option is Retain existing SLAs of the cloud server.

        If you choose Retain existing SLAs of the cloud server when modifying an SLA associated with the project, the cloud server SLA will be replaced when the SLA associated with the cloud server is the same as that associated with the project.

      2. You can select one or more of the following options for SLA Application Policy:
        • Apply the project/resource set SLA to all cloud servers that are not associated with any SLA.
        • Apply the project/resource set SLA to the created cloud server.
      3. Specify the filtering mode for the cloud servers to be filtered in the protection plan. You can select Including and Excluding, Include only, or Excluding to filter protected cloud servers.

        You can use cloud server names or keywords in those names to exclude or include some cloud servers. Use commas (,) to separate multiple cloud servers.

        Exact match and fuzzy match of cloud server names are supported. Fuzzy match rules include prefix match (for example, abc*), suffix match (for example, *abc), and middle match (for example, *abc*).

    6. Click OK.

      If the current system time is later than the first backup start time of the SLA, you can select to perform backup immediately after the backup SLA is associated or wait for the next scheduling based on the backup policy set in the SLA.

  3. Periodically back up a cloud server or cloud server group.

    1. Click the Cloud Server or Cloud Server Groups tab and select the cloud server or cloud server group to be protected.
    2. Click Protect.
    3. Determine whether to enable Automatic Disk Protection.

      This step is required only for periodic backup of cloud servers.

      Automatic Disk Protection is enabled by default to back up all disks of a cloud server. You can disable Automatic Disk Protection and click Please select to manually select desired cloud disks for backup.

      Automatic disk protection is enabled by default. After this function is enabled, all disks of the cloud server, including newly detected disks, are protected. After automatic disk protection is disabled, you need to manually select disks for protection.

      If system disks are excluded, generated copies do not support file-level restoration.

    4. Select a preset or user-defined SLA.

      You can also click Create to create an SLA.

    5. Select the agent host.

      If the agent host is not selected, the system performs backup by selecting one or multiple agent hosts from the agent hosts configured during resource registration.

    6. Set Auto Indexing of Backup Copies. (This parameter is supported only in 1.6.0 and later versions.)
      If a VM or VM group is selected for protection and the disks to be backed up contain the system disk, the system automatically creates an index for the backup copy after this function is enabled. When using the copy for restoration, you can select one or more files in the copy for restoration and search for files, directories, or links in the copy.

      Indexes occupy backup storage space. If the number of index files in the system exceeds 1 billion, perform operations by referring to "Failed to Create a Copy Index" in the OceanProtect Appliance 1.5.0-1.6.0 Troubleshooting.

    7. Click OK.

      If the current system time is later than the first backup start time of the SLA, you can select to perform backup immediately after the backup SLA is associated or wait for the next scheduling based on the backup policy set in the SLA.

  4. Optional: Perform manual backup.

    If you want to execute a backup job immediately, perform manual backup through the following operations. Otherwise, skip this step.

    1. Click the Project or Cloud Server tab based on the type of the data to be backed up.

      On the Project tab page, when a project is selected to perform backup, only cloud servers associated with the same SLA as the project are backed up.

    2. Choose More > Manual Backup on the right of the row where the desired project or cloud server resides.

      You can select multiple resources to perform manual backup in batches. Select multiple resources and choose More > Manual Backup in the upper left corner of the page.

    3. Set the name of the copy generated during manual backup.

      If this parameter is left unspecified, the system sets the name to backup_Timestamp by default.

    4. Select a protection policy, which can be full backup or forever incremental (synthetic full) backup.

      For 1.6.0 and later versions: If the selected protection policy is different from that configured in the associated SLA, the WORM configuration does not take effect.

    5. Click OK.
