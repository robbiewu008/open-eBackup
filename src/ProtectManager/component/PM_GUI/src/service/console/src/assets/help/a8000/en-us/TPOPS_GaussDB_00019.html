Step 7: Performing Backup

Before performing backup, you need to associate the resources to be protected with a specified SLA policy. The system protects the resources based on the SLA policy and periodically performs backup jobs based on the SLA policy. You can perform a backup job immediately through manual backup.

Prerequisites

  • Before initiating backup, ensure that the database is in the normal state.
  • If the agent host has been replaced before the backup is initiated, scan for resources again. For details about how to scan for resources, see Managing GaussDB Projects.
  • Before initiating backup, ensure that the default backup root path of the instance has been configured on the OLTP OPS management page. For details about how to configure the default backup root path, see Step 3: Configuring the Default Backup Root Path for the Instance on the TPOPS Management Page.
  • Before initiating backup, ensure that monitoring has been enabled for the instances managed by OLTP OPS. For details about how to enable monitoring, see Step 4: Enabling Instance Monitoring on the TPOPS Management Page.
  • Before initiating backup, ensure that backUpAgent has been started on the agent host. For details about how to start backUpAgent, see "Starting backUpAgent" in the OceanProtect Appliance 1.5.0-1.6.0 ProtectAgent Installation Guide.
  • Before initiating backup, ensure that the XBSA backup whitelist has been enabled on TPOPS. For details about how to enable the whitelist, see Step 2: Enabling the XBSA Backup Whitelist on the TPOPS Node.
  • When a full backup is performed for a newly created instance for the first time, ensure that no backup job is being executed on the instance. Log in to the TPOPS management page and choose Task Center > Task Details to view task details.
  • If the database kernel version is earlier than 505.1.1, ensure that the replica consistency protocol of the instance is Quorum. Otherwise, log backup will fail.

Context

  • During manual backup, parameters (except Automatic Retry) defined in the SLA are applied to the manual backup job.
  • Copies generated by manual backup are retained for the duration defined in the SLA.

    If the protection policy selected for a manual backup job is not defined in the SLA, the generated backup copy is retained permanently. If full backup is not defined in the SLA but selected during manual backup, the copies generated by the manual full backup are retained permanently.

  • If a replication or archive policy has been defined in the SLA, the system will perform replication or archiving once based on the SLA when you perform manual backup.

Precautions

For versions earlier than 1.5.0.SPC3, there is a hole area between the log backup and the next data backup. Data files generated in this area will not be backed up. For 1.5.0.SPC3 and later versions, there is no such a hole area. However, log backup cannot be performed for restored instances. For example, if instance A is restored to instance B, log backup cannot be performed for instance B.

Procedure

  1. Choose Protection > Databases > GaussDB.

    For 1.5.0, choose Protection > Cloud Platforms > GaussDB.

  2. Click the Instance tab.
  3. In the row that contains the target instance, choose More > Protect.

    You can also select multiple unprotected instances and click Protect for batch protection.

  4. Select an SLA.

    You can also click Create to create an SLA.

    If a WORM policy has been configured for the resource to be protected, select an SLA without a WORM policy to create protection to avoid WORM policy conflicts.

  5. Click OK.

    If the current system time is later than the first backup start time of the SLA, you can select to perform backup immediately after the backup SLA is associated or wait for the next scheduling based on the backup policy set in the SLA.

  6. Optional: Perform manual backup.

    If you want to execute a backup job immediately, perform manual backup through the following operations. Otherwise, skip this step.

    1. In the row of the target instance, choose More > Manual Backup.

      You can select multiple resources to perform manual backup in batches. Select multiple resources and choose More > Manual Backup in the upper left corner of the page.

    2. Set the name of the copy generated through manual backup.

      If this parameter is left empty, the system sets the name to backup_Timestamp by default.

    3. Select a protection policy: Full backup, differential backup, or Log Backup.

      For 1.6.0 and later versions: If the selected protection policy is different from that configured in the associated SLA, the WORM configuration does not take effect.

    4. Click OK.
