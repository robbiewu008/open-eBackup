Overview of User Roles

To prevent mis-operations from compromising service system stability and service data security, you can define user roles to control user permissions. Before using this document, ensure that you have the operation permissions of the current account.

Definition of User Roles

Role: defines the objects that a user operates and accesses.

The system provides only built-in roles. Built-in roles are preset in the system and have specific permissions. Table 1 lists the permissions that built-in roles have.

Table 1 Built-In Role

Preset Role

Permission

Built-in System User

System Administrator

Users with this role have all permissions on the system.

NOTE:

After system administrator creation, you need to set the sender email address and password retrieval email address.
  • sysadmin: a built-in system administrator who can log in to the GUI and invoke REST APIs.

    For the default password, see the Account List.

  • cluster_admin: a machine-machine account of the built-in system administrator. This account cannot be used to log in to the GUI and can only invoke REST APIs.

    The password is randomly generated during system initialization. You can reset the password on the product.

Data Protection Administrator

Users with this role have data protection permissions, such as backup and restoration.

NOTE:
  • Data protection administrators can only view the permissions of this role and change their own passwords. They cannot view or change the passwords of other users.
  • Data protection administrators do not have permissions to create and delete multi-cluster members, add and delete HAs, and view node details.
  • Data protection administrators do not have permissions to view information of Security Policies, Certificates, and Data Security.
  • SAML data protection administrators cannot view the information of System Capacity, Data Reduction, and Performance. You can view backup and archive quota information on the home page.

mmdp_admin: a machine-machine account of the built-in data protection administrator. This account cannot log in to the GUI and can only invoke REST APIs.

The password is randomly generated during system initialization. You can reset the password on the product.

Auditor

Users with this role can audit the system with the read-only permission.

mm_audit: a machine-machine account of the built-in auditor. This account cannot log in to the GUI and can only invoke REST APIs.

The password is randomly generated during system initialization. You can reset the password on the product.

Remote Device Administrator

Users with this role can be used for authentication and authorization between the source and target clusters during copy replication.

-

Disaster Recovery Administrator

Users with this role have the permission to query cluster and capacity information and perform operations on SAML users, such as adding, deleting, modifying, and querying SAML users, and managing quotas and functions.

-

Device administrator

Internal role, which cannot be configured on the GUI. This role is used for upgrade and O&M of the product. For example, this role can be used to upgrade the product through SmartKit.

admin: a built-in device administrator who can log in to the device management GUI and invoke some REST APIs. Initialize the password upon the first login. For details about how to change the password, see the Account List.

Querying Permissions of the Current Login User

After logging in to the system, you can perform the following operations to learn about the operation permissions and scope of your current account.

Procedure

  1. Choose System > Security > Users and Roles > Users.
  2. (Optional) Click next to Role to filter users. You can also click next to Name to search for a user.
  3. In the user list, view the Role information of the current login user and determine the operation permissions and scope of the current account according to Table 1.

    If you log in as a system administrator, you can view information about all users of the current device.
