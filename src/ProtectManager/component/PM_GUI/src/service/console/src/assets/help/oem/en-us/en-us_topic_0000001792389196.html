Live Mounting the Oracle Database

This section describes how to live mount a database copy to a new location for quickly starting a new database.

Context

  • A backup copy, replication copy, or copy generated by live mount can be used for live mount.
  • The copy cannot be mounted to the original location.

Prerequisites

ProtectAgent and other related software have been installed on the target host for mounting.

For details, see the ProtectAgent Installation Guide.

Procedure

  1. If a database with the same name exists, shut down the database before mounting a backup copy. Otherwise, the mount will fail. To shut down the database, do as follows:

    1. Log in to the Oracle database host.

      Use PuTTY to log in to the Oracle database host and run the su - oracle command to switch to user oracle.

    2. Run the following commands to log in to the Oracle database instance.
      • OS authentication:

        export ORACLE_SID=Database instance name

        sqlplus / as sysdba

      • Database authentication:

        export ORACLE_SID=Database instance name

        sqlplus

        Enter the name of a user with the sysdba permissions as prompted:

        username as sysdba

        Enter the password as prompted.

    3. Run the following command to shut down the database with the same name:

      shutdown immediate;

  2. Choose Explore > Live Mount > Oracle.
  3. Click Create.
  4. On the Select Resource tab, select the source database corresponding to the copy to be mounted and click Next.

    • If the resource type is cluster, only the management IP address of the cluster is displayed, while the members in the cluster are not displayed.
    • If a host or cluster is offline, it will not be live mounted. Offline hosts or clusters are not displayed.

  5. On the Select Copy tab, do as follows:

    1. Select the copy to be mounted.

      The copy to be live mounted must be in the Normal state.

    2. Select a mount update policy.

      If the mount update policy is not selected in this step, you can configure the mount update policy by modifying mount information later.

    3. Click Next.

  6. On the Mount Options tab page, set mount parameters, and click Next.

    1. Set the basic and advanced parameters for live mount.
      Table 1 describes the related parameters.
      • You can set either Bandwidth (MB/s) or Standard IOPS (8 KB) or both of them. You are advised to set only Bandwidth (MB/s) for bandwidth-intensive services and only Standard IOPS (8 KB) for IOPS-intensive services. If you cannot determine the service type or the service type changes frequently, you can set both parameters and the system will control the traffic using the parameter with a smaller value.
      • You can query the conversion between the standard IOPS (8 KB) model and real I/O model in the conversion table on the management page.
      • The maximum traffic of mounted copies is limited to ensure the performance of backup and restoration jobs. In addition, the minimum traffic of mounted copies is set to ensure the minimum performance of mounted copies.

      Enter a script name that ends with .sh.       Modify Database Settings

      Modify the database parameters based on the actual configuration of the host to which the database is to be mounted. If any parameter is set incorrectly, the database may fail to be mounted.

      When the copy of the Oracle cluster is mounted to the ASM of the Oracle single-node system, change the value of *.cluster_database to false.

      NOTE:

      During restoration, ensure that the free memory size of the target host must be greater than the total size of the SGA and PGA of the database. Otherwise, the restoration may fail.

      1. Check the free memory size of the target host.

        For details, see 2.

      2. After enabling Modify Database Settings, search for *.pga_aggregate_target and *.sga_target parameters.

        Check whether the free memory size is greater than the sum of *.pga_aggregate_target and *.sga_target values.

        • If yes, no further action is required.
        • If no, go to 3.
      3. Modify either *.pga_aggregate_target or *.sga_target parameters or both of them to ensure that the free memory size is greater than the sum of *.pga_aggregate_target and *.sga_target values.

        If multiple identical parameters are found, modify all of them.

      4. If *.pga_aggregate_target is modified, change Instance name.__pga_aggregate_target to the following:

        Instance name.__pga_aggregate_target = *.pga_aggregate_target * 1024 *1024

        If *.sga_target is modified, change Instance name.__sga_target to the following:

        Instance name.__sga_target = *.sga_target * 1024 *1024
    2. Click Next.

  7. Preview the mount parameters. After confirming that the parameters are set correctly, click Finish.

    • To remotely access the Oracle database that is live mounted, you need to set the listening parameters of the database. For example, you need to configure the database listening parameters before anonymizing data in the copies generated during live mount.

      For details, see Configuring Oracle Database Listening Parameters.

    • If a copy is used for live mount immediately after it is generated, the live mount job may fail because the copy is still in the initialization state. In this case, try again in 3 minutes.
    • If the target database version is later than the copy database version, database upgrade is automatically triggered during live mount. Ensure that the archive log path of the target database has available space. Otherwise, mount jobs cannot be executed.
