Restoring a SQL Server Database

This section describes how to use a copy to restore a database that has been backed up to the original or different host.

Prerequisites

  • Supported resource types include the single instance, cluster instance, availability group, and database.
  • Before restoring data to a different host, ensure that a database instance exists on the target host.
  • Before restoration, ensure that the remaining space of the data directory at the target location for restoration is greater than the size of the copy used for restoration before reduction. Otherwise, restoration will fail.
  • The target location must have a reserved capacity greater than three times the size of the copy to be used for restoration.

Procedure

  1. Choose Explore > Copy Data > Databases > SQL Server.
  2. You can search for copies by database resource or copy. This section describes how to search for copies by resource.

    On the Resources tab page, find the resource to be recovered by resource name and click the resource name.

  3. Select a resource for copy-based restoration.
  4. Specify a copy or a point in time for restoration.

    You can specify a copy or any point in time between two copies for restoration.

    For the OceanProtect E6000 Appliance, when a replication copy is used for restoration, other copies of the resource cannot be replicated at the remote end.
    • Restoration using a specified copy
      1. Select the year, month, and day in sequence to find the copy.

        If is displayed under a month or date, a copy exists in the month or on the date.

      2. On the Copy Data page, click and choose More > Restore in the row where the desired copy is located.
    • Restoration to a specified time point
      1. Select the year, month, and day in sequence to find the copy.

        If is displayed under a month or date, a copy exists in the month or on the date.

      2. On the copy data page, click . In the blue part on the timeline, select a copy and choose More > Restore.

  5. Table 1 describes the parameters for restoring an SQL Server database.

    Table 1 Parameters of SQL Server database restoration

    Parameter

    Description

    Restore To

    Select either Original location or New location.

    Location

    If Restore To is set to Original location, the original location is displayed by default.

    Tag

    NOTE:

    This parameter is available only in 1.6.0 and later versions.

    If you set Restore To to New location, you can filter the resources to be restored by tag.

    Target Host/Cluster

    If Restore To is set to New location, select the original host or a different host for restoration.

    Target Instance

    If Restore To is set to New location, select the instance on the target host.

    NOTE:
    • This parameter is displayed when Type is set to Databases.
    • Ensure that databases or availability groups with the same name do not exist in the target instance.

    Target Path

    NOTE:

    This parameter is supported only in version 1.5.0.

    Path of the target host to which database files are restored. This parameter is displayed when Restore To is set to New location.

    NOTE:
    • This parameter is displayed when Type is set to Databases.
    • Ensure that drive letter of the target path instead of the drive letter root directory exists and the database user has the write permission on the target path.
    • Ensure that no database file with the same name exists in the target path.

    Rename Database

    After this function is enabled, you can rename the restored database.

    NOTE:
    • This parameter is displayed when Type is set to Databases.
    • If this function is enabled and the database is restored to the original location, select the owning instance of the database and scan for resources after the restoration completes (on the Protection > Databases > SQL Server > Instances page, locate the row that contains the target instance, and choose More > Resource Scan), and perform a full backup for the database.

    Script to Run Before Restoration

    You can execute a custom script before a restore job is executed and after it succeeds or fails based on your need.

    Enter a script name that ends with .bat. Ensure that the script is stored in the DataBackup\ProtectClient\ProtectClient-E\bin\thirdparty directory on the database host and the user (user root by default) who has logged in to the database host is permitted to execute the script.

    NOTE:
    • This parameter is displayed when Type is set to Databases.
    • If Script to Run upon Restoration Success is configured, the status of the restoration job is displayed as Succeeded on the OceanProtect Appliance even if the script fails to be executed. Check whether the job details contain information indicating that a post-processing script fails to be executed. If yes, modify the script in a timely manner.
