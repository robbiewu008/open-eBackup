Step 3: Creating a Backup SLA

Customize an SLA to protect resources based on your requirements.

Context

For details about customizing an SLA, see About SLA.

Procedure

  1. Choose Protection > Protection Policies > SLAs.
  2. Click Create.
  3. Enter an SLA name. The default name is SLA_Current timestamp.
  4. Select an application type for the SLA to be created.

    1. Click Applications and select the application for the SLA to be created.
      • General SLA

        A general SLA can be applied to resources except filesets, SQL Server, PostgreSQL, Kingbase, Redis, ClickHouse, MongoDB, NAS shares, NAS file systems, Huawei Cloud Stack GaussDB, TDSQL, Kubernetes CSI, TiDB, volumes, common shares, GaussDB T single-node systems, and object storage.

      • Specific to application

        Select an application type of the SLA to be created, for example, NAS file system. This section describes how to create Specific to application.

    2. Click OK.

  5. Configure a backup policy.

    1. Click the Backup Policy icon.
    2. Configure basic parameters for a backup policy.

      Table 1 describes the related parameters.

      Set the backup frequency, copy retention period, and backup time window based on service requirements. The recommended settings are as follows:
      • The backup interval must be longer than the backup duration.
      • The retention period must be longer than the backup interval.
      Table 1 Basic parameters of a backup policy

      Parameter

      Description

      Forever Incremental (Synthetic Full) Backup

      Name

      Name of the policy.

      -
      Configure the first execution time, backup interval, copy retention period, and backup time window for forever incremental (synthetic full) backup. The initial backup is a full backup by default.
      • Configure the backup frequency and copy retention period for forever incremental (synthetic full) backup.
        • By Year

          Configure the job to be executed once every year on xx (month) xx (day). Set copies to be retained for xx days, weeks, months, years, or permanently.

          If the date does not exist in the year, no copy is generated.

        • By Month

          Configure the job to be executed on xx (day) (multiple days can be selected) or the last day of each month. Set copies to be retained for xx days, weeks, months, years, or permanently.

          If the backup job is set to be executed on the xxth day of each month, no copy is generated when the date does not exist in the current month.

        • By Week

          Configure the job to be executed every Monday, Tuesday, Wednesday, Thursday, Friday, Saturday, or Sunday (you can select multiple options). Set copies to be retained for xx days, weeks, months, years, or permanently.

        • By Day

          Configure the job to be executed every xx days starting from xx (year) xx (month) xx (day). Configure copies to be retained for xx days, weeks, months, years, or permanently.

        • By Hour

          Configure the job to be executed every xx hours starting from xx (year) xx (month) xx (day). Configure copies to be retained for xx days, weeks, months, years, or permanently.

        • By Minute

          Configure the job to be executed every xx minutes starting from xx (year) xx (month) xx (day). Configure copies to be retained for xx days, weeks, months, years, or permanently.

      • Set the allowed time period for executing Forever Incremental (Synthetic Full) Backup. Forever Incremental (Synthetic Full) Backup jobs will not be scheduled at the time beyond the time period.
        NOTE:
        • If the end time is earlier than or the same as the start time, the end time is actually the end time of the next day.
        • If the backup job is not completed within the specified time window, the system does not stop the backup job, but reports an event.
        • Once the retention period expires, the system automatically deletes the expired copies.

      A maximum of four forever incremental (synthetic full) backup policies can be configured for an SLA. The name of each forever incremental (synthetic full) backup policy can be customized.

    3. Configure advanced parameters for the backup policy.
      Table 2 describes the related parameters.
      Table 2 Advanced parameters of a backup policy

      Parameter

      Description

      Specify Target Location
      If you have deployed the cluster HA feature, specify the target storage location for the backup copy. If no target location is specified, the system automatically selects a backup target storage unit through intelligent balancing.
      • Backup storage unit: corresponds to a backup cluster node. After a backup storage unit is selected, copies are stored to the storage unit. If the capacity of the storage unit is insufficient, the backup job will fail.
      • Backup storage unit group: is a collection of backup storage units. After a backup storage unit group is selected, the system automatically selects a target backup storage unit based on the storage policy of the storage unit group.

      Rate Limiting Policies

      Average maximum amount of data that can be transmitted per second in a backup job.

      Select a rate limiting policy created in Step 2: (Optional) Creating a Rate Limiting Policy.

      Automatic Indexing

      After this function is enabled, you can search for files, directories, or links in copies.

      Target Deduplication

      After this function is enabled, the amount of data before copy data reduction increases. Exercise caution when performing this operation.
      Job Timeout Alarm
      NOTE:

      This parameter is available only in 1.6.0 and later versions.

      After this function is enabled, an alarm is sent if the job execution time exceeds the time window. The alarm needs to be manually cleared.

      Job Failure Alarm

      After this function is enabled, an alarm is automatically sent if a job fails and is automatically cleared after successful execution.

      Automatic Retry

      After this function is enabled, the system automatically retries if a periodic backup job fails.

      The number of retries ranges from 1 to 5, and the wait time ranges from 1 to 30 minutes. For example, if the number of retries is set to 3 and the wait time is set to 5 minutes, the system retries for a total of three times at an interval of 5 minutes.
      NOTE:
      • During automatic retry, the system creates a backup job. The backup job is executed only when it is within the preset time window.
      • This function takes effect only during periodic backup. Automatic retry is not supported if a backup job fails to be manually initiated.
    4. Click OK.

  6. Click OK.
