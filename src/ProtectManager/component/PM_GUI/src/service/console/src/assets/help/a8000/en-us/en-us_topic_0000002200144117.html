GaussDB Archive Space Full and Subsequent Backup Failures Due to Long Delay in Manual Archiving (Log Backup) After Enabling the Archiving Function (Applicable to OLTP OPS)

Symptom

If manual archiving (log backup) is not performed for a long time after the archiving function is enabled for GaussDB, subsequent backup will fail.

Possible Causes

In GaussDB 2.23.01.275 and earlier, if manual archiving (log backup) is not performed for a long time due to GaussDB defects, the log archiving slot will be fully occupied and subsequent log archiving requests will fully occupy the connection pool. As a result, subsequent backup fails.

Troubleshooting

  1. Ensure that the log archive slot is fully occupied.

    1. Log in to an instance node as user root. For a centralized instance, log in to the active DN (usually dn_6001). For a distributed instance, log in to the first CN (usually cn_5001). Access $GAUSSLOG/gs_log/Node name.
    2. Search for BarrierArchMain will in the kernel logs generated around the time when the error is reported.
    3. Check the thread IDs in sequence. If the length is 599999 field is found in the thread ID attachment, the log archive slot is full. For details, see 2. If the length is 599999 field is not found, contact technical support engineers.

  2. Perform a full backup again after restarting the instance.
