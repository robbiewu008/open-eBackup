Restoring MySQL/MariaDB Databases

This section describes how to use a copy to restore a database that has been backed up to the original or a different host.

Prerequisites

  • Before executing the restoration job, manually stop the MySQL/MariaDB service.
  • Before restoration, ensure that the remaining space of the data directory at the target location for restoration is greater than the size of the copy used for restoration before reduction. Otherwise, restoration will fail.

Procedure

  1. Choose Explore > Copy Data > Databases > MySQL/MariaDB.
  2. Search for copies by MySQL/MariaDB resource or copy. This section describes how to search for copies by resource.

    On the Resources tab page, find the resource to be restored by resource name and click the resource name.

  3. Select the year, month, and day in sequence to find the copy.

    If is displayed below a month or date, copies exist in the month or on the day.

  4. Choose More > Restore in the row where the copy is located.

    Table 1 describes the related parameters.

    Table 1 Parameters for restoring a MySQL/MariaDB database

    Parameter

    Description

    Restore To

    Select either Original location or New location.

    Location

    If you select Original location, the target location is displayed by default.

    Target Instance

    If you select restoration to New location, select the target instance for restoration.

    Target Database

    If you select restoration to New location, select the target database for restoration.

    Rename Database

    If you select restoration to New location, you can rename the to-be-restored database.

    Forcible Restoration

    This function is disabled by default. After this function is enabled, data can be forcibly restored when the database page is damaged.

    Script to Run Before Restoration

    You can execute a custom script as required before a restoration job is executed or after it succeeds or fails.

    Enter the absolute path and name of the script and ensure that the user who logs in to the database host has the execute permission on the script.

    Non-Windows OSs support .sh scripts, for example, /opt/script.sh.

    NOTE:

    If Script to Run upon Restoration Success is configured, the status of the restoration job is displayed as Successful on the product even if the script fails to be executed. Check whether the job details contain a message indicating that a post-processing script fails to be executed. If yes, modify the script in a timely manner. (For other applications involving post-processing scripts, use the same source here.)

    After the restoration is complete, wait for several minutes until data synchronization between nodes is complete.
