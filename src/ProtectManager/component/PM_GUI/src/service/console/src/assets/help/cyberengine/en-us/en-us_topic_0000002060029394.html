Failed to Log In to the OceanCyber WebUI

Symptom

Failed to log in to the OceanCyber WebUI.

Possible Causes

System management data is damaged or lost, such as system configurations, database data, and internal certificates. As a result, the protectmanager-system-base container is abnormal and the WebUI cannot be logged in to.

Troubleshooting

  1. Log in to the background as user kadmin and switch to user root.
  2. Run the following command to back up and restore management data.

    • Local system management data restoration
      sh /opt/k8s/chart/install_script/sysbackup_recovery.sh

      This command is used to obtain the local management data backup list for restoration.

    • Remote system management data restoration
      sh /opt/k8s/chart/install_script/sysbackup_recovery.sh sftp_user sftp_host sftp_path sftp_filename

      This command is used to obtain the backup data from the remote SFTP server for restoration.

      Table 1 Parameters for restoring remote system management data

      Parameter

      Description

      sftp_user

      (Mandatory) Username for logging in to the SFTP server.

      sftp_host

      (Mandatory) Host IP address of the SFTP server.

      sftp_path

      (Optional) Path of the SFTP server. The default value is /var/sysbackup.

      sftp_filename

      (Optional) Name of the SFTP server files. The default value is the latest file name in the path where the SFTP server is located.

  3. After the script is successfully executed, run the following command to view the pod named protectmanager-system-base-0 in the namespace and wait until the value of READY in the third column is 2/2:

    kubectl get pod -A -owide

  4. Log in to the OceanCyber WebUI again and check whether the login is successful.

    • If yes, no further action is required.
    • If no, reinstall the environment or keep the fault environment intact and contact technical support engineers.

Suggestion

  1. Before running the script, ensure that the infrastructure pod status is normal.
  2. Before running the script, ensure that the backup package is the backup package of the current environment (not reinstalled environment or other environments).
  3. Before running the script, ensure that the backup package version is the same as the current environment version.
