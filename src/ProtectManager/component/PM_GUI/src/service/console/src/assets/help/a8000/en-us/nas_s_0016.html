Step 5: Creating a Backup SLA

Customize an SLA to protect resources based on your requirements.

Context

For details about customizing an SLA, see About SLA. For details about the constraints, see Constraints.

Procedure

  1. Choose Protection > Protection Policies > SLAs.
  2. Click Create.
  3. Enter an SLA name. The default name is SLA_Current timestamp.
  4. Select an application type for the SLA to be created.

    1. Click Applications and select the application for the SLA to be created.
      • General SLA

        A general SLA can be applied to resources except filesets, SQL Server, PostgreSQL, Kingbase, Redis, ClickHouse, MongoDB, NAS shares, NAS file systems, Huawei Cloud Stack GaussDB, TDSQL, Kubernetes CSI, TiDB, volumes, common shares, Active Directory, GaussDB T single-node systems, object storage, GaussDB (1.6.0 and later versions), and SAP HANA (as an application) (1.6.0 and later versions).

      • Specific to application

        Select an application type for the SLA to be created, for example, NAS file system. This section assumes that an SLA of the Specific to application type is created.

    2. Click OK.

  5. Configure a backup policy.

    1. Click the Backup Policy icon.
    2. Configure basic parameters for a backup policy.

      Table 1 describes the related parameters.

      Set the backup interval, copy retention period, and backup time window based on service requirements. The recommended settings are as follows:
      • The backup interval must be longer than the backup duration.
      • The retention period must be longer than the backup interval.
      • At least one backup policy must be created. To add another backup policy, click on the right of the page.
      • A maximum of four backup policies can be added for each backup type.
      Table 1 Basic parameters of a backup policy

      Parameter

      Description

      Forever Incremental (Synthetic Full) Backup

      Name

      Name of the policy.

      -
      Configure the first execution time, backup interval, copy retention period, and backup time window for forever incremental (synthetic full) backup. The first backup is a full backup by default.
      • Configure the backup interval and copy retention period for forever incremental (synthetic full) backup.
        • By Year

          Configure the job to be executed once every year on xx (month) xx (day). Set copies to be retained for xx days, weeks, months, years, or permanently.

          If the date does not exist in the year, no copy is generated.

        • By Month

          Configure the job to be executed on xx (day) (multiple days can be selected) or the last day of each month. Set copies to be retained for xx days, weeks, months, years, or permanently.

          If the backup job is set to be executed on the xxth day of each month, no copy is generated when the date does not exist in the current month.

        • By Week

          Configure the job to be executed every Monday, Tuesday, Wednesday, Thursday, Friday, Saturday, or Sunday (you can select multiple options). Set copies to be retained for xx days, weeks, months, years, or permanently.

        • By Day

          Configure the job to be executed every xx days starting from xx (year) xx (month) xx (day). Configure copies to be retained for xx days, weeks, months, years, or permanently.

        • By Hour

          Configure the job to be executed every xx hours starting from xx (year) xx (month) xx (day). Configure copies to be retained for xx days, weeks, months, years, or permanently.

        • By Minute

          Configure the job to be executed every xx minutes starting from xx (year) xx (month) xx (day). Configure copies to be retained for xx days, weeks, months, years, or permanently.

      • Set the time period for executing forever incremental (synthetic full) backup. The forever incremental (synthetic full) backup job will not be scheduled beyond the set time range.
        NOTE:
        • If the end time is earlier than or the same as the start time, the end time is actually the end time of the next day.
        • If the backup job is not completed within the specified time window, the system does not stop the backup job, but reports an event.
        • Once the retention period expires, the system automatically deletes the expired copies.
    3. For 1.6.0 and later versions: To generate a WORM copy, enable WORM and set the WORM validity period. Otherwise, skip this step.
      For WORM Validity Period, you can select Same as the copy retention period or Custom validity period.
      • WORM copies cannot be deleted or their validity periods cannot be shortened.
      • The custom WORM validity period cannot be longer than the copy retention period.
      • A WORM copy cannot be changed to a non-WORM copy within its validity period.
    4. Configure advanced parameters for the backup policy.
      Table 2 describes the related parameters.
      Table 2 Advanced parameters of a backup policy

      Parameter

      Description

      Specify Target Location

      NOTE:

      This parameter is supported only by the X series backup appliances.
      Specify the target storage location of backup copies.
      • Backup storage unit: corresponds to a backup cluster node. After a backup storage unit is selected, copies are stored to the storage unit. If the capacity of the storage unit is insufficient, the backup job will fail.
      • Backup storage unit group: is a collection of backup storage units. After a backup storage unit group is selected, the system automatically selects a target backup storage unit based on the storage policy of the storage unit group.
      NOTE:
      • For 1.5.0, this parameter is displayed only when you have deployed the cluster HA feature. You can leave the target location unspecified. If you do not specify the target location, the system automatically selects the target storage unit through intelligent balancing.
      • For 1.6.0 and later versions, you must specify the target location.

      Rate Limiting Policies

      Average maximum amount of data that can be transmitted per second in a backup job.

      Select the rate limiting policy created in (Optional) Step 4: Creating a Rate Limiting Policy.

      Automatic Indexing

      NOTE:

      This parameter is available only in 1.5.0.
      After this function is enabled, you can search for files, directories, or links in copies.
      NOTE:

      Indexes occupy backup storage space. If the number of index files in the system exceeds 1 billion, perform operations by referring to "Failed to Create a Copy Index" in the OceanProtect Appliance 1.5.0-1.6.0 Troubleshooting.

      Target Deduplication

      NOTE:

      This parameter is supported only by the X series backup appliances.

      When the average file size is greater than or equal to 32 KB, you are advised to enable this function to save backup storage space.

      After target deduplication is enabled, service data is deduplicated before being written to the storage space, saving storage space and improving backup efficiency.
      Standard SmartCompression
      NOTE:

      After standard SmartCompression is enabled, service data is compressed before being written to the storage space, saving storage space and improving backup efficiency.
      Job Timeout Alarm
      NOTE:

      This parameter is available only in 1.6.0 and later versions.

      After this function is enabled, an alarm is sent if the job execution time exceeds the time window. The alarm needs to be manually cleared.

      Job Failure Alarm

      After this function is enabled, an alarm is automatically sent if a job fails and is automatically cleared after successful execution.

      Automatic Retry

      After this function is enabled, the system automatically retries if a periodic backup job fails.

      The number of retries ranges from 1 to 5, and the wait time ranges from 1 to 30 minutes. For example, if the number of retries is set to 3 and the wait time is set to 5 minutes, the system retries for a total of three times at an interval of 5 minutes.
      NOTE:
      • During automatic retry, the system creates a backup job. The backup job is executed only when it is within the preset time window.
      • This function takes effect only during periodic backup. Automatic retry is not supported if a backup job fails to be manually initiated.
    5. Click OK.

  6. (Optional) Configure archive policies.

    The built-in storage space of the OceanProtect is limited. If you need to retain copy data for a long time, you are advised to archive the copy data to external storage.

    You can configure the archive policies by clicking Archive Policy or by modifying the SLA.

    For details about how to configure the archive policies, see Step 2: Creating an Archive SLA for Backup Copies.

  7. (Optional) Configure replication policies.

    The OceanProtect allows you to replicate local data to a remote data center. In case a disaster occurs, copy data in the remote data center can be used to restore production data and services can be taken over, achieving remote data protection.

    You can configure the replication policies by clicking Replication Policy or by modifying the SLA.

    For details about how to configure the replication policies, see Step 7: Creating a Replication SLA.

    For 1.6.0 and later versions:

    • The WORM validity period of the backup policy in the SLA must be shorter than the copy retention period in the replication policy.
    • If WORM is not configured at the local end but configured at the target end, the WORM validity period is the same as the replication copy retention period.
    • If WORM is configured at the local end but not at the target end, the WORM validity period of replication copies is the same as that of WORM copies at the local end. The retention period of replication copies is greater than or equal to the WORM validity period of backup copies at the local end.
    • If WORM is configured at both the local and target ends and WORM is configured by resource in the WORM configuration of Data Security at the target end, all replication copies will have WORM enabled, and the WORM validity period is the same as the retention period of replication copies.
    • At the local end, if you configure backup copies to be retained permanently and set WORM Validity Period to Same as the copy retention period, the WORM validity period is permanent. After a backup copy is generated, if you modify the SLA policy (by adding a replication policy and setting Replication and Retention Rules to Replicate all copies), the WORM validity period at the target end is the same as the retention period of replication copies.

  8. Click OK.
