Step 4: Creating a Backup SLA

You can customize SLAs based on your requirements to protect resources.

Procedure

  1. Choose Protection > Protection Policies > SLAs.
  2. Click Create.
  3. Enter an SLA name.
  4. Select an application type for the SLA to be created.

    1. Click the Applications icon to create a general SLA or an SLA for a specified application.
      • On the General SLA tab page, click All Resources.

        A general SLA can be applied to resources except filesets, SQL Server, PostgreSQL, Kingbase, Redis, ClickHouse, MongoDB, NAS shares, NAS file systems, Huawei Cloud Stack GaussDB, TDSQL, Kubernetes CSI, TiDB, volumes, common shares, Active Directory, GaussDB T single-node systems, object storage, GaussDB (1.6.0 and later versions), GaussDB (DWS), SAP HANA (as an application in 1.6.0 and later versions), SAP on Oracle, AntDB, VMware, and Hyper-V.

      • On the Specific to application tab page, select an application type.

        This section describes how to create an SLA for a specified application.

      Only OceanProtect X series backup appliances support NAS file systems.

    2. Click OK.

  5. Configure a backup policy.

    1. Click the Backup Policy icon.
    2. Set basic parameters of the backup policy.

      Table 1 describes the related parameters.

      Set the backup interval, copy retention period, and backup time window based on service requirements. The recommended settings are as follows:
      • The backup interval must be longer than the backup duration.
      • The retention period of a full backup copy must be longer than that of an incremental backup (or forever incremental (synthetic full) backup) copy.
      • At least one backup policy must be created. To add more backup policies, click on the right of the page.
      • A maximum of four backup policies can be added for each backup type.
      Table 1 Basic parameters of a backup policy

      Parameter

      Description

      Full Backup

      Name

      Set the backup policy name.

      -
      Configure the backup interval, copy retention period, and backup time window for full backup.
      • Configure the backup interval and copy retention period for full backup.
        • By Year

          Configure the job to be executed once every year on xx (month)-xx (day). Configure copies to be retained for xx days, weeks, months, years, or permanently.

          If the date does not exist in the year, no copy is generated.

        • By Month

          Configure the job to be executed on xx (day) (multiple days can be selected) or the last day of each month. Configure copies to be retained for xx days, weeks, months, years, or permanently.

          If the backup job is set to be performed on the xxth day of each month, no copy is generated when the date does not exist in the current month.

        • By Week

          Configure the job to be executed every Monday, Tuesday, Wednesday, Thursday, Friday, Saturday, or Sunday (you can select multiple options). Configure copies to be retained for xx days, weeks, months, years, or permanently.

        • By Day

          Configure the job to be executed every xx days starting from xx (year)-xx (month)-xx (day). Configure copies to be retained for xx days, weeks, months, years, or permanently.

        • By Hour

          Configure the job to be executed every xx hours starting from xx (year)-xx (month)-xx (day). Configure copies to be retained for xx days, weeks, months, years, or permanently.

      • Set the time period for performing full backup. The full backup job will not be scheduled beyond the time range.
        NOTE:
        • If the end time is earlier than or the same as the start time, the end time is actually the end time of the next day.
        • If the backup job is not completed within the specified time window, the system does not stop the backup job, but reports an event.
        • Once the retention period expires, the system automatically deletes the expired copies.

      Forever Incremental (Synthetic Full) Backup

      Name

      Set the backup policy name.

      -
      Configure the first execution time, backup interval, copy retention period, and backup time period for forever incremental (synthetic full) backup.
      • Configure the backup interval and copy retention period for forever incremental (synthetic full) backup.
        • By Year

          Configure the job to be executed once every year on xx (month) xx (day). Set copies to be retained for xx days, weeks, months, years, or permanently.

          If the date does not exist in the year, no copy is generated.

        • By Month

          Configure the job to be executed on xx (day) (multiple days can be selected) or the last day of each month. Set copies to be retained for xx days, weeks, months, years, or permanently.

          If the backup job is set to be executed once on the xxth day of each month, no copy is generated when the date does not exist in the current month.

        • By Week

          Configure the job to be executed every Monday, Tuesday, Wednesday, Thursday, Friday, Saturday, or Sunday (you can select multiple options). Set copies to be retained for xx days, weeks, months, years, or permanently.

        • By day

          Configure the job to be executed every xx days starting from xx (year) xx (month) xx (day). Set copies to be retained for xx days, weeks, months, years, or permanently.

        • By Hour

          Configure the job to be executed every xx hours starting from xx (year) xx (month) xx (day). Set copies to be retained for xx days, weeks, months, years, or permanently.

        • By Minute

          Configure the job to be executed every xx minutes starting from xx (year) xx (month) xx (day). Set copies to be retained for xx days, weeks, months, years, or permanently.

      • Set the time period for executing forever incremental (synthetic full) backup. The forever incremental (synthetic full) backup job will not be scheduled beyond the set time range.
        NOTE:
        • If the end time is earlier than or the same as the start time, the end time is actually the end time of the next day.
        • If the backup job is not completed within the specified time window, the system does not stop the backup job, but reports an event.
        • Once the retention period expires, the system automatically deletes the expired copies.
    3. To generate WORM copies, enable the WORM function and set the WORM validity period. Otherwise, skip this step.
      For WORM Validity Period, you can select Same as the copy retention period or Custom validity period.
      • Only the OceanProtect X series backup appliances and OceanProtect E1000 support WORM.
      • WORM copies cannot be deleted or their validity periods cannot be shortened.
      • The custom WORM validity period cannot be longer than the copy retention period.
      • A WORM copy cannot be changed to a non-WORM copy within its validity period.
      • If you select Same as the copy retention period for WORM Validity Period and choose to retain copies permanently, the actual WORM validity period is 20 years for the copies in snapshot format and 70 years for the copies in directory format. This is because copies in snapshot format can be retained for up to 20 years and those in directory format can be retained for up to 70 years on the backup storage.
    4. Set other parameters of the backup policy.
      Table 2 describes the related parameters.
      Table 2 Other parameters of a backup policy

      Parameter

      Description
      Specify Target Location
      NOTE:

      Only the OceanProtect X series backup appliances and OceanProtect E1000 support this parameter.
      Specify the target storage location of backup copies.
      • Backup Storage Units: A backup storage unit corresponds to a backup cluster node. After a backup storage unit is selected, copies are stored to the storage unit. If the capacity of the storage unit is insufficient, the backup job will fail.

        For details about how to create a backup storage unit, see "Managing Backup Storage Units" in the OceanProtect DataBackup 1.5.0-1.6.0 Cluster HA Feature Guide.

      • Backup Storage Unit Groups: A backup storage unit group is a collection of backup storage units. After a backup storage unit group is selected, the system automatically selects a target backup storage unit based on the storage policy of the storage unit group.

        For details about how to create a backup storage unit group, see "(Optional) Creating Backup Storage Unit Groups" in the OceanProtect DataBackup 1.5.0-1.6.0 Cluster HA Feature Guide.

      Rate Limiting Policy

      Maximum amount of data that can be transmitted by a backup job per second.

      Choose a created rate limiting policy.
      Source Deduplication
      NOTE:

      Only the OceanProtect X series backup appliances and OceanProtect E1000 (with the OceanProtect used as backup storage) support this parameter.
      After this function is enabled, the amount of data to be transmitted can be reduced, but more client resources will be occupied.
      NOTE:
      • A client can only be linked with one storage unit. To enable source deduplication, select Backup Storage Unit for Specify Target Location. Otherwise, a storage unit that is not linked with the client may be selected through intelligent balancing. As a result, source deduplication fails to be enabled. If different protected objects are associated with different SLAs, where different backup storage units are specified, select different clients for the protected objects. Otherwise, source deduplication will fail to be enabled for some backup jobs.
      • If a client has been linked with a storage unit but is not performing a backup job with source deduplication enabled, you can run the dataturbo delete storage_object storage_name=? command on the host where the client is deployed to delete the source deduplication link with the storage unit. The value of storage_name in the command can be queried by running the dataturbo show storage_object command. This way, when the client performs a backup job with source deduplication enabled, the link between the client and a new storage unit can be established.
      • Visit OceanProtect Compatibility Query to check whether source deduplication is supported by the OS of the host where the client is deployed. If not supported, source deduplication will not take effect after being configured.
      • If a built-in client (that is, the client is co-deployed with the OceanProtect) is used for backup, source deduplication does not take effect after being configured.

      Job Timeout Alarm

      After this function is enabled, an alarm is sent if the job execution time exceeds the time window. The alarm needs to be manually cleared.

      Job Failure Alarm

      After this function is enabled, an alarm is sent when a job fails. The alarm is automatically cleared when the next job is successful.

      Automatic Retry

      After this function is enabled, the system automatically retries if a backup job fails.

      The number of retries ranges from 1 to 5, and the waiting time ranges from 1 to 30 minutes. For example, if the number of retries is set to 3 and the waiting time to 5 minutes, the system makes a total of three retries and performs each retry every 5 minutes.
      NOTE:

      During automatic retry, the system creates a backup job. The backup job is executed only when it is within the preset time window.
    5. Click OK.

  6. (Optional) Configure an archive policy.

    The built-in storage space of the OceanProtect is limited. If you need to retain copy data for a long time, you are advised to archive the copy data to external storage.

    You can configure an archive policy by clicking the Archive Policy icon or by modifying the SLA.

    For details about how to configure an archive policy, see Archiving.

    For OceanProtect E1000, if the device type of the backup storage unit or backup storage unit group specified in the backup policy is NormalServer, no archive policy can be created.

  7. (Optional) Configure a replication policy.

    The OceanProtect allows you to replicate local data to a remote data center. If a disaster occurs, copy data in the remote data center can be used to restore production data and services can be taken over.

    You can configure a replication policy by clicking the Replication Policy icon or by modifying the SLA.

    For details about how to configure a replication policy, see "Creating a Replication SLA" for the corresponding product model in Replication.

    • The WORM validity period of the backup policy in the SLA must be shorter than the copy retention period in the replication policy.
    • If WORM is not configured at the local end but configured at the target end, the WORM validity period is the same as the replication copy retention period.
    • If WORM is configured at the local end but not at the target end, the WORM validity period of replication copies is the same as that of WORM copies at the local end. The retention period of replication copies is greater than or equal to the WORM validity period of backup copies at the local end.
    • If WORM is configured at both the local and target ends and WORM is configured by resource in the WORM configuration of Data Security at the target end, all replication copies will have WORM enabled, and the WORM validity period is the same as the replication copy retention period.
    • At the local end, if you configure local backup copies to be retained permanently and set WORM Validity Period to Same as the copy retention period, the WORM validity period is permanent. After a backup copy is generated, if you modify the SLA policy (by adding a replication policy and setting Replication and Retention Rules to Replicate all copies), the WORM validity period at the target end is the same as the retention period of replication copies.

  8. Click OK.
