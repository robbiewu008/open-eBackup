In GaussDB (DWS) 8.0.0, Restoration to a New Cluster Using a Cluster Copy Fails Because the Node that Delivers the Restoration Command Fails to Copy the cluster_static_config File

Fault Symptom

In GaussDB (DWS) 8.0.0, if a cluster copy fails to be used for the restoration to a new cluster, the message "the Roach command fails to be executed" is displayed in the job details.

Possible Cause

The metadata of the GaussDB (DWS) node that delivers the restoration command has been successfully restored. However, an error occurs when the cluster_static_config file is copied to another node. As a result, the data restoration fails.

Suggestion

  1. Use PuTTY to log in to the CN in the GaussDB (DWS) cluster where the client has been deployed.
  2. Run the following commands in sequence to switch to the cluster user and load environment variables:

    su - Cluster username
    source Environment variable file path

  3. Run the cd $GAUSSLOG/ roach/controller/ command to go to the log file directory and check whether the log file whose name starts with roach_controller contains the following error information:

    cluster_static_config: No such file or directory

    Example:

    • If error information is displayed, perform the restoration job again.
    • If no error information is contained, go to 4.

  4. Log in to other CNs in the GaussDB (DWS) cluster where the client has been deployed and perform 2 to 3. If no node contains the error information, collect logs and contact Huawei technical support engineers.
