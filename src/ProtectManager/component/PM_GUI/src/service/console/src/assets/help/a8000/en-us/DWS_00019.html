Step 6: Performing Backup

Before performing backup, you need to associate the resources to be protected with a specified SLA policy. The system protects the resources based on the SLA policy and periodically performs backup jobs based on the SLA policy. You can perform a backup job immediately through manual backup.

Precautions

  • Copies generated by manual backup are retained for the period defined in the SLA.

    If the protection policy selected during manual backup is not defined in the SLA, the generated backup copy is retained permanently. If incremental backup is not defined in the SLA but is selected during manual backup, the copies generated by the manual incremental backup are retained permanently.

  • If a replication or archive policy has been defined in the SLA, the system will perform replication or archiving once based on the SLA when you perform manual backup.
  • During manual backup, the parameters defined in the SLA (except Automatic Retry), such as Rate Limiting Policies, are applied to the manual backup job.

Procedure

  1. Choose Protection > Big Data > GaussDB (DWS).
  2. Click the Cluster, Schema Sets, or Table Sets tab, locate the row that contains the resource to be protected, and choose More > Protect.

    You can also select multiple resources for batch protection.

    To protect a table set:

    • Ensure that GDS has been started on the agent host. GDS running on the GaussDB (DWS) node occupies resources in the production environment. Therefore, you are advised to start GDS on the agent host. For details about how to start GDS, see "Installing, Configuring, and Starting GDS" in the GaussDB (DWS) Product Documentation of the corresponding version.
    • Parameters -d, -p, and -t are required for starting GDS. The directory for storing data of each GDS must be unique. The following is an example:
      /home/gds_user/bin/gds -d /home/gds_user/input_data_1/ -p 192.168.97.240:9001 -H 0/0 -l /home/gds_user/gds_log/gds_process1.log -D -t 2
/home/gds_user/bin/gds -d /home/gds_user/input_data_2/ -p 192.168.97.240:9002 -H 0/0 -l /home/gds_user/gds_log/gds_process2.log -D -t 2
/home/gds_user/bin/gds -d /home/gds_user/input_data_3/ -p 192.168.97.240:9003 -H 0/0 -l /home/gds_user/gds_log/gds_process3.log -D -t 2
/home/gds_user/bin/gds -d /home/gds_user/input_data_4/ -p 192.168.97.240:9004 -H 0/0 -l /home/gds_user/gds_log/gds_process4.log -D -t 2
    • If you start GDS on multiple agent hosts, the system uses the agent host with the largest number of GDSs for backup. Therefore, you are advised to start multiple GDSs as one user on only one agent host. To achieve the highest performance, it is recommended that the number of GDSs to be started be four times and eight times the number of backup storage units for a dual-controller system and a four-controller system, respectively.
    • If the GaussDB (DWS) node is deployed with the sandbox, map GDS configuration directory in the sandbox to the corresponding directory on the host by referring to Precautions for GDS Running in a GaussDB (DWS) Node Where the Sandbox Is Deployed before GDS-based table set backup.

  3. Enter Backup Metadata Storage Path. Enter a directory on a GaussDB (DWS) node to store backup copy metadata. If the directory does not exist, the system will automatically create it. The GaussDB (DWS) cluster user must have the read and write permissions on the directory.

    • At least 20 GB disk space must be reserved on the Backup Metadata Storage Path. Do not configure the path as a system volume or production service volume. Otherwise, production services will be affected when the disk space is insufficient.
    • To check whether the GaussDB (DWS) cluster user has the read and write permissions on the backup metadata storage path, log in to the GaussDB (DWS) node as user root and run the ls -ld Backup metadata storage path command. If you do not have the read and write permissions, run the sudo chown GaussDB (DWS) cluster user:Backup metadata storage path of the GaussDB (DWS) cluster user command to change the permissions.

      An example is as follows:

      ls -ld /home/omm/backup
      sudo chown omm:omm /home/omm/backup

  4. Select an SLA.

    You can also click Create to create an SLA.

    Preset SLAs cannot be selected when schema sets or table sets are to be protected.

  5. Click OK.

    If the current system time is later than the first backup start time of the SLA, you can select to perform backup immediately after the backup SLA is associated or wait for the next scheduling based on the backup policy set in the SLA.

  6. Optional: Perform manual backup.

    If you want to execute a backup job immediately, perform manual backup through the following operations. Otherwise, skip this step.

    1. In the row where the cluster, schema set, or table set to be backed up resides, choose More > Manual Backup.
    2. Set the name of the copy generated through manual backup.

      If this parameter is left unspecified, the system sets the name to backup_Timestamp by default.

    3. Set a protection policy.
      • For clusters, you can select Full backup, Incremental backup, or Differential Backup.
      • For table sets, only Full backup is supported.
      • For schema sets, only Full backup is supported. For GaussDB (DWS) 8.2.1 and later versions, Incremental backup is also supported for schema sets.
    4. Click OK.

      You can select multiple resources to perform manual backup in batches. Select multiple resources and choose More > Manual Backup.
